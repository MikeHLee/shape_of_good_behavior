# Streamlit App V2: The Hodge Navigator

## 1. Conceptual Reversal

The V1 app was "Reward First": define a reward function, then see how agents behave.
The V2 app is **"Behavior First"**: observe/generate behavior, then deduce the topological structure of the reward manifold.

## 2. User Experience Flow

The application is divided into three distinct stages, mirroring the new experimental protocol.

### Stage 1: The Semantic Explorer (Episode Generation)
*Goal: Explore the state space and generate raw trajectories.*

**UI Components:**
*   **Left Panel (The "World"):** Displays the current **Semantic State Description** (e.g., "The server is responding with 500 errors...").
*   **Center Panel (The "Agent"):** Displays the **MCP Tool Calls** generated by the small LLM (e.g., `check_logs()`, `restart_service()`).
*   **Right Panel (The "Oracle"):** A chat interface where the user (acting as the Environment/Oracle) types what happens next based on the agent's action.
    *   *Auto-Mode Option:* Use a larger LLM (e.g., GPT-4) to simulate the environment response automatically.

**Interaction:**
*   Step through an episode: State -> Agent Action -> Environment Response -> New State.
*   Repeat for $N$ episodes to build a "cloud" of trajectories.

### Stage 2: The Hodge Critic (Feedback & Projection)
*Goal: Assign topological gradients to the raw behavior.*

**UI Components:**
*   **Main View (The "Manifold"):** A 2D/3D Plotly scatter plot showing the embedded states from Stage 1.
    *   Points are colored by their "potential" (initially flat/random).
*   **Feedback Interface:**
    *   Select a trajectory or a specific transition on the plot.
    *   **Slider (0-1):** "Progress/Helpfulness."
    *   **Text Input:** "Critique" (e.g., "Good, but unsafe.").
*   **Real-time Visualization:**
    *   As the user provides feedback, arrows (vectors) appear on the plot.
    *   "Critique" text is embedded and projected as a directional vector (e.g., "unsafe" might point away from a high-reward cluster).

### Stage 3: Manifold Stitching (Analysis)
*Goal: Resolve inconsistencies and visualize the "True" reward surface.*

**UI Components:**
*   **Topological Dashboard:**
    *   **H1 Indicator:** A gauge showing the magnitude of the "Curl" (inconsistency).
    *   **"Stitch" Button:** Triggers the Hodge Decomposition.
*   **Post-Stitch View:**
    *   The vector field on the plot is "cleaned" (curl removed).
    *   The underlying surface is re-contoured to show the **Gradient Flow** (the consistent policy).
    *   **Black Holes:** Regions with negative feedback loops are highlighted as "Event Horizons."

## 3. Technical Implementation Details

### Backend
*   **State Management:** Session state stores the graph of visited Semantic States.
*   **Embeddings:** `sentence-transformers` (e.g., `all-MiniLM-L6-v2` for speed, or `UAE-Large-V1` for quality) to embed state descriptions and critiques on the fly.
*   **Dimensionality Reduction:** `UMAP` (parametric if possible) to project high-dim embeddings to 3D.
*   **Topological Compute:** `numpy` / `scipy` for basic vector field operations (Hodge decomposition on a discrete graph/simplicial complex).

### Frontend (Streamlit)
*   **Plotly:** Interactive 3D scatter plots for the manifold.
*   **Streamlit Chat:** For the Agent/Oracle interaction.

## 4. Development Roadmap

1.  **Skeleton**: Build the Stage 1 "Semantic Explorer" loop (State -> Action -> State).
2.  **Embedding Pipeline**: Integrate the embedding model to visualize states as they are generated.
3.  **Feedback Loop**: Add the ranking slider and critique input.
4.  **Math Engine**: Implement the discrete Hodge decomposition for the graph.
5.  **Visualization**: Connect the math engine to the Plotly surface rendering.
